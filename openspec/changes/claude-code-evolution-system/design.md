## Context

Claude Code 是一个基于 CLI 的 AI 编程助手,目前所有配置都依赖静态的 `~/.claude/CLAUDE.md` 文件。用户需要手动维护这些配置,且无法从历史会话中自动学习。

**当前痛点:**
- 用户需要重复告诉 Claude 相同的偏好
- 相同问题会被反复解决,没有知识积累
- 有价值的工作模式无法沉淀为可复用技能
- 配置文件随着时间变得臃肿且难以维护

**参考系统:**
- **OpenClaw**: 实现了 Memory + Skills + SOUL 三位一体的自进化机制
- **claude-mem**: 提供了基于 MCP 的持久化记忆存储和语义检索

**技术约束:**
- 必须保持与现有 Claude Code CLI 的兼容性
- 自动化操作需要用户明确授权
- Token 使用需要控制在合理范围
- 系统应该轻量级,不影响正常使用性能

## Goals / Non-Goals

**Goals:**
- 自动从会话历史中学习用户偏好和工作模式
- 动态生成和更新 CLAUDE.md 配置文件
- 识别重复模式并自动创建可复用技能
- 提供可视化界面查看和管理学习结果
- 保持系统轻量级,后台运行不影响主流程

**Non-Goals:**
- 不修改 Claude Code 核心代码(通过插件/Hook 方式集成)
- 不实现完全自动化(关键决策需要用户确认)
- 不支持多用户协作(单用户本地系统)
- 不实现实时学习(使用定时任务周期性分析)
- 不处理 Claude Code 以外的数据源

## Decisions

### 决策 1: 采用定时任务 + MCP 架构

**选择**: 使用 node-cron 定时任务 + claude-mem MCP 服务的架构

**理由:**
- **定时任务优于实时学习**: 避免干扰用户正常使用,可在空闲时运行
- **MCP 优于自建存储**: claude-mem 已经提供了成熟的向量检索和持久化能力
- **松耦合优于紧耦合**: 独立进程不影响 Claude Code 稳定性

**替代方案:**
- ❌ **Claude Code Hook 实时触发**: 会增加每次对话的延迟,影响用户体验
- ❌ **自建 SQLite + Chroma**: 重复造轮子,维护成本高
- ❌ **OpenClaw 完整移植**: 过于复杂,不符合 Claude Code 的简洁哲学

### 决策 2: 分阶段学习策略

**选择**: 采用 3 层学习策略:
1. **观察期** (1-3 天): 收集数据但不做修改
2. **建议期** (第 4-7 天): 生成建议但需用户确认
3. **自动期** (第 7 天后): 高置信度的更新可自动应用

**理由:**
- 避免初期误判导致错误配置
- 给用户足够时间理解和信任系统
- 渐进式增强用户体验

**替代方案:**
- ❌ **立即自动化**: 风险太高,用户接受度低
- ❌ **永远需要确认**: 失去自动化价值,用户疲劳

### 决策 3: 模块化 MD 配置结构

**选择**: 拆分 CLAUDE.md 为源文件 + 动态拼接的架构

```
~/.claude-evolution/
├── source/              # 手动编辑的源文件
│   ├── CORE.md         # 核心行为准则 (用户维护)
│   ├── STYLE.md        # 沟通风格 (用户维护)
│   └── CODING.md       # 编程规范 (用户维护)
├── learned/             # 自动学习的片段
│   ├── preferences.md  # 用户偏好 (系统生成)
│   ├── workflows.md    # 工作流模式 (系统生成)
│   └── solutions.md    # 常见问题解决方案 (系统生成)
└── output/
    └── CLAUDE.md        # 最终拼接输出
```

**理由:**
- 清晰分离手动配置和自动学习内容
- 用户可随时覆盖自动学习的结果
- 支持版本控制和差异对比

**替代方案:**
- ❌ **单文件追加**: 难以区分来源,维护困难
- ❌ **完全自动化**: 用户失去控制权

### 决策 4: LLM 驱动的经验提取

**选择**: 使用 Claude API 进行会话摘要和模式识别

**提示词策略:**
```typescript
const extractPrompt = `
分析以下会话记录,提取:
1. 用户明确表达的偏好 (语气、风格、工具选择)
2. 重复出现的问题和解决方案 (3次以上)
3. 工作流模式 (固定操作序列)
4. 需要改进的配置点

输出 JSON 格式,包含置信度分数。
`;
```

**理由:**
- Claude 擅长理解自然语言中的隐含偏好
- 可以识别复杂的模式关联
- 输出结构化数据便于后续处理

**替代方案:**
- ❌ **规则引擎**: 无法处理自然语言的语义
- ❌ **本地小模型**: 理解能力不足,误判率高

### 决策 5: 技能生成采用模板驱动

**选择**: 预定义技能模板 + 动态填充的方式

**技能类型模板:**
1. **Command Skill** - 封装命令序列
2. **Pattern Skill** - 识别并应用代码模式
3. **Workflow Skill** - 多步骤工作流

**理由:**
- 结构化模板保证生成质量
- 降低 LLM 生成错误的概率
- 用户易于理解和修改

**替代方案:**
- ❌ **完全 LLM 生成**: 质量不稳定,格式难以保证
- ❌ **只记录不生成**: 失去自动化价值

### 决策 6: 安全迁移现有配置

**选择**: 采用"安全迁移 + 渐进式接管"的双重保障策略

**关键问题**: 用户可能已有精心配置的 `~/.claude/CLAUDE.md`,直接替换会导致配置丢失。

**解决方案**:

**阶段 1: 检测和备份**
```typescript
// 初始化时检测现有配置
if (exists('~/.claude/CLAUDE.md')) {
  // 1. 创建永久备份
  backup('~/.claude/CLAUDE.md.backup-YYYY-MM-DD');

  // 2. 提供三个选项
  const choice = await prompt([
    'A. 导入现有配置 (推荐)',
    'B. 保留原配置 (独立运行)',
    'C. 取消安装'
  ]);
}
```

**阶段 2: 智能导入(选项 A)**
```typescript
// 解析现有配置并拆分到 source/ 目录
const parsed = parseExistingCLAUDE(content);
// 识别区块: 核心理念 → CORE.md
//           沟通风格 → STYLE.md
//           编程规范 → CODING.md
// 无法分类 → UNMAPPED.md (用户手动整理)

// 显示拆分结果
console.log(`
✓ 已识别并导入:
  - 核心理念 → source/CORE.md
  - 沟通风格 → source/STYLE.md
  - 编程规范 → source/CODING.md

⚠️  以下内容无法自动分类,已保存到:
  ~/.claude-evolution/source/UNMAPPED.md
  请手动查看并整合到对应文件中。
`);
```

**阶段 3: 独立运行(选项 B)**
```
不修改任何现有文件:
~/.claude/CLAUDE.md                      ← 用户原配置 (仍在使用)
~/.claude-evolution/output/CLAUDE.md    ← 我们的配置 (试验)

用户可以:
- 查看新配置: cat ~/.claude-evolution/output/CLAUDE.md
- 对比差异: diff ~/.claude/CLAUDE.md ~/.claude-evolution/output/CLAUDE.md
- 验证后手动切换: claude-evolution switch --enable
```

**阶段 4: 渐进式切换**
```bash
# 查看当前状态
$ claude-evolution status
当前模式: 独立运行
原配置: ~/.claude/CLAUDE.md (使用中)
新配置: ~/.claude-evolution/output/CLAUDE.md (待切换)

# 对比差异
$ claude-evolution diff
显示两个配置的差异...

# 确认无问题后切换
$ claude-evolution switch --enable
✓ 已创建备份
✓ 已创建软链接
✓ 切换完成

# 如果有问题,随时回滚
$ claude-evolution switch --disable
✓ 已恢复原配置
```

**多重安全保障**:
1. **永久备份**: `CLAUDE.md.backup-YYYY-MM-DD` 永不删除
2. **操作日志**: 记录所有文件修改操作
3. **回滚命令**: `claude-evolution rollback --to-original`
4. **干运行模式**: `claude-evolution init --dry-run` 预览操作

**理由:**
- 绝不在用户不知情的情况下覆盖配置
- 提供多个选项,用户保持完全控制权
- 渐进式切换降低风险,随时可回滚
- 即使导入失败,原配置仍然安全

**替代方案:**
- ❌ **直接替换**: 风险极高,用户配置丢失
- ❌ **追加模式**: 文件臃肿,难以管理
- ❌ **强制迁移**: 用户体验差,缺乏信任

### 决策 7: 使用 TypeScript + Node.js 实现

**选择**: TypeScript 作为主要实现语言

**技术栈:**
- **运行时**: Node.js 18+
- **语言**: TypeScript 5+
- **调度**: node-cron
- **MCP 客户端**: @modelcontextprotocol/sdk
- **LLM**: @anthropic-ai/sdk
- **配置验证**: zod
- **文件监听**: chokidar
- **CLI**: commander
- **UI**: 可选使用 ink (React for CLI)

**理由:**
- 与 Claude Code 主体技术栈一致
- TypeScript 提供类型安全
- Node.js 生态成熟,依赖丰富
- 可以直接读取 Claude Code 的配置和数据

**替代方案:**
- ❌ **Python**: 不利于与 Claude Code 集成
- ❌ **Go**: 开发效率低,生态不如 Node.js

## Risks / Trade-offs

### 风险 1: Token 消耗成本

**风险**: 定期 LLM 调用会产生 API 费用

**缓解措施:**
- 使用 Haiku 模型进行分析(成本仅 Opus 的 1/20)
- 增量分析而非全量重复分析
- 用户可配置分析频率和深度
- 提供本地缓存减少重复调用

### 风险 2: 错误学习导致配置污染

**风险**: 误判用户意图,生成错误配置

**缓解措施:**
- 3 阶段学习策略,初期不自动应用
- 所有自动更新都记录到日志,支持回滚
- 用户可随时查看和撤销学习结果
- 设置置信度阈值,低置信度内容不自动应用

### 风险 3: 隐私和安全

**风险**: 会话数据可能包含敏感信息

**缓解措施:**
- 默认过滤敏感模式(API keys, passwords, tokens)
- 用户可配置黑名单关键词
- 本地处理,不上传到第三方服务
- 提供数据清理工具

### 风险 4: 用户现有配置被覆盖

**风险**: 初始化时直接替换用户精心配置的 CLAUDE.md,导致配置丢失

**缓解措施:**
- **永久备份**: 初始化前自动创建 `CLAUDE.md.backup-YYYY-MM-DD`
- **用户选择**: 提供"导入"、"独立运行"、"取消"三个选项
- **智能拆分**: 尝试自动解析并拆分到 source/ 目录
- **渐进式切换**: 默认独立运行,用户验证后手动切换
- **完整回滚**: `claude-evolution switch --disable` 随时恢复原配置
- **干运行模式**: `--dry-run` 预览操作不实际执行
- **操作日志**: 记录所有文件修改,可追溯

→ **详见决策 6: 安全迁移现有配置**

### 风险 5: 系统复杂度增加

**风险**: 新增组件可能导致维护负担

**缓解措施:**
- 模块化设计,每个组件独立可测试
- 完善的错误处理和日志
- 提供降级方案(关闭自动学习仍可正常使用)
- 文档齐全,包括故障排查指南

### 权衡 1: 实时性 vs 资源消耗

**选择**: 牺牲实时性,使用定时任务

**权衡点:**
- ✅ 不影响用户正常使用性能
- ✅ 可在低峰期运行
- ❌ 学习结果有延迟(最多 24 小时)

### 权衡 2: 自动化程度 vs 用户控制

**选择**: 半自动化,关键决策需确认

**权衡点:**
- ✅ 用户保持最终控制权
- ✅ 建立信任,降低误判风险
- ❌ 初期需要较多用户参与

### 权衡 3: 功能完整性 vs 系统复杂度

**选择**: MVP 先实现核心功能,UI 可选

**权衡点:**
- ✅ 快速验证核心价值
- ✅ 降低开发和维护成本
- ❌ 初期用户体验可能较简陋

## Migration Plan

### 阶段 1: 初始化 (首次运行)

1. 检测 `~/.claude-evolution/` 是否存在
2. 如不存在,运行初始化向导:
   ```bash
   npx claude-evolution init
   ```
3. 向导步骤:
   - 询问是否授权访问 claude-mem
   - 配置分析频率 (每日/每 12 小时/手动)
   - 迁移现有 CLAUDE.md 到 source/ 目录
   - 生成默认配置文件

### 阶段 2: 观察期 (1-3 天)

1. 定时任务仅收集数据,不生成建议
2. 后台构建用户画像和模式库
3. 向用户展示收集到的数据统计

### 阶段 3: 建议期 (4-7 天)

1. 开始生成配置建议
2. 通过 CLI 提示用户查看建议:
   ```bash
   claude-evolution review
   ```
3. 用户逐条确认或拒绝

### 阶段 4: 自动期 (7 天后)

1. 高置信度建议自动应用
2. 低置信度建议仍需确认
3. 用户可随时查看进化日志

### 回滚策略

**场景 1: 配置错误**
```bash
# 回滚到上一个版本
claude-evolution rollback --version 1

# 回滚到特定日期
claude-evolution rollback --date 2024-03-10
```

**场景 2: 完全重置**
```bash
# 清除所有学习数据,保留源文件
claude-evolution reset --keep-source

# 完全重置
claude-evolution reset --full
```

**备份机制:**
- 每次更新前自动备份当前 CLAUDE.md
- 保留最近 30 天的备份
- 备份存储在 `~/.claude-evolution/backups/`

- 方案 B: 预留兼容性接口

### Q4: 是否需要 Web UI?

**问题**: CLI 是否足够,还是需要图形界面?

**待讨论方案:**

## Data Flow

```
1. 用户使用 Claude Code
   ↓
2. 会话数据自动存储到 claude-mem
   ↓
3. 定时任务触发 (每 24 小时)
   ↓
4. Analyzer 从 claude-mem 读取近期会话
   ↓
5. Extractor 使用 LLM 提取经验
   {
     preferences: ["简洁回复", "避免emoji"],
     patterns: [
       { name: "git提交流程", frequency: 5 }
     ],
     solutions: [
       { problem: "TS类型错误", solution: "使用zod" }
     ]
   }
   ↓
6. Learners 处理提取结果
   - PreferenceLearner → learned/preferences.md
   - SkillDetector → learned/skills.md
   - WorkflowLearner → learned/workflows.md
   ↓
7. MD Generator 拼接生成 CLAUDE.md
   source/CORE.md
   + source/STYLE.md
   + learned/preferences.md
   → output/CLAUDE.md
   ↓
8. 软链接到 ~/.claude/CLAUDE.md
```
