## 新增需求

### 需求：分析完成通知
系统必须在分析流程完成时通过 WebSocket 推送通知到所有连接的客户端。

#### 场景：推送分析完成事件
- **当** 分析流程成功完成
- **则** 向所有连接的 WebSocket 客户端发送"analysis_complete"事件
- **则** 事件数据包含完成时间、新建议数量
- **则** 前端显示桌面通知（如已启用）

#### 场景：推送分析失败事件
- **当** 分析流程执行失败
- **则** 向客户端发送"analysis_failed"事件
- **则** 事件数据包含错误信息摘要

### 需求：新建议通知
系统必须在生成新建议时推送通知。

#### 场景：推送新建议事件
- **当** 分析器生成新的待审批建议
- **则** 发送"new_suggestions"事件
- **则** 事件数据包含新建议的类型和数量
- **则** 前端 Dashboard 自动刷新待审批数量

### 需求：建议状态变更通知
系统必须在建议被批准或拒绝时推送通知到其他客户端。

#### 场景：推送建议批准事件
- **当** 用户通过 API 批准建议
- **则** 向其他连接的客户端发送"suggestion_approved"事件
- **则** 其他客户端的 Review 页面从列表移除该建议

#### 场景：推送建议拒绝事件
- **当** 用户拒绝建议
- **则** 向其他客户端发送"suggestion_rejected"事件
- **则** 其他客户端同步更新列表

### 需求：系统级桌面通知
系统必须在配置启用时通过守护进程发送操作系统原生桌面通知。

#### 场景：显示分析完成通知
- **当** 分析流程成功完成且通知已启用
- **则** 守护进程通过 node-notifier 发送系统通知
- **则** 通知标题为"claude-evolution"
- **则** 通知内容为"分析完成！3 条新建议待审核"
- **则** 通知包含点击跳转 URL: http://localhost:10010/review

#### 场景：点击通知打开网页
- **当** 用户点击桌面通知
- **则** 系统打开默认浏览器
- **则** 浏览器打开新标签页（如果已有标签页不受影响）
- **则** 导航到通知指定的目标页面

#### 场景：根据通知类型跳转
- **当** 通知类型为 analysis_complete
- **则** 跳转到 /review 页面
- **当** 通知类型为 analysis_failed
- **则** 跳转到 /dashboard 页面
- **当** 通知类型为 new_suggestions
- **则** 跳转到 /review 页面
- **当** 通知类型为 config_changed
- **则** 跳转到 /settings 页面
- **当** 通知类型为 system_error
- **则** 跳转到 /dashboard 页面

### 需求：跨平台通知支持
系统必须在不同操作系统上使用原生通知机制。

#### 场景：macOS 系统通知
- **当** 运行在 macOS 上
- **则** 使用 macOS Notification Center 显示通知
- **则** 通知显示在屏幕右上角
- **则** 点击通知打开 URL 使用默认浏览器

#### 场景：Windows 系统通知
- **当** 运行在 Windows 上
- **则** 使用 Windows Toast 通知显示
- **则** 通知显示在屏幕右下角
- **则** 点击通知打开 URL 使用默认浏览器

#### 场景：Linux 系统通知
- **当** 运行在 Linux 上
- **则** 使用 notify-send 或 libnotify 显示通知
- **则** 点击通知打开 URL 使用默认浏览器
- **则** 如果系统未安装通知工具，在日志中记录警告

### 需求：连接状态指示
前端必须显示 WebSocket 连接状态。

#### 场景：显示连接状态
- **当** WebSocket 连接成功建立
- **则** 页面右上角显示绿色"已连接"指示器

#### 场景：显示断开状态
- **当** WebSocket 连接断开
- **则** 页面显示黄色"正在重连"指示器
- **则** 前端自动尝试重新连接

#### 场景：重连失败提示
- **当** 重连尝试 5 次后仍失败
- **则** 显示红色"连接失败"指示器
- **则** 提示用户检查服务器状态
