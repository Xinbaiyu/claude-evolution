## 新增需求

### 需求：LLM 合并前按相似度分组观察

系统必须在将观察数据发送给 LLM 进行合并处理之前，使用 Jaccard 词相似度和观察类型对其进行聚类分组。相似度 > 0.5 且 `type` 字段相同的观察必须通过 Union-Find 聚类放入同一组。

#### 场景：高相似度的观察被分到同一组
- **当** 两条观察具有相同的 `type` 且 `calculateSimilarity()` 返回值 > 0.5
- **则** 它们必须被放入同一组

#### 场景：低相似度的观察保留在不同组
- **当** 两条观察的 `calculateSimilarity()` ≤ 0.5 或 `type` 值不同
- **则** 它们必须保留在不同的组中

#### 场景：传递相似性分组
- **当** 观察 A 与 B 相似，B 与 C 相似，但 A 与 C 不直接相似
- **则** 三者（A、B、C）必须通过 Union-Find 传递性放入同一组

### 需求：强制执行最大组大小

系统必须将超过 15 条观察的任何组拆分为不超过 15 条的子组，按得分（confidence × mentions）排序，最高分优先。

#### 场景：组超过大小限制
- **当** 一个相似度组包含 20 条观察
- **则** 系统必须将其拆分为两个子组：一个包含 15 条观察（最高分），另一个包含 5 条观察

#### 场景：组在大小限制内
- **当** 一个相似度组包含 10 条观察
- **则** 系统必须将其作为单个组保留，不进行拆分

### 需求：单例组跳过 LLM 处理

系统不得对仅包含一条观察的组进行 LLM 调用。这些观察必须直接返回，不做任何修改。

#### 场景：组中只有单条观察
- **当** 聚类后一个组只包含 1 条观察
- **则** 该观察必须被包含在最终结果中，无需任何 LLM 调用

#### 场景：所有观察都是唯一的
- **当** 没有任何观察与其他观察的相似度 > 0.5
- **则** 不得进行任何 LLM 调用，所有观察必须原样返回

### 需求：有限并发处理各组

系统必须通过并行 LLM 调用处理多观察组，最大并发数为 3 个同时请求。

#### 场景：多组并行处理
- **当** 有 5 个组需要 LLM 处理
- **则** 最多 3 个组同时处理，其余 2 个排队等待

#### 场景：单个组
- **当** 只有 1 个组需要 LLM 处理
- **则** 它必须立即处理，无并发开销

### 需求：按组错误隔离

系统必须在组级别处理 LLM 故障。如果某组的 LLM 调用失败，该组中的观察必须原样返回（不合并），而其他组继续正常处理。

#### 场景：一个组失败，其他组成功
- **当** A 组的 LLM 调用失败，而 B 组和 C 组成功
- **则** A 组的观察必须原样返回（串联），B 组和 C 组返回合并后的结果

#### 场景：所有组都失败
- **当** 所有 LLM 组调用都失败
- **则** 所有观察必须原样返回（等同于 `fallbackNoMerge`）
